# Longevity & Wellness Library — Maintenance Guide

This file tells Claude how to add and maintain content in this site.

## Project structure

```
aging-library/
├── index.html          ← browse page (catalog is embedded here by build.py)
├── catalog.json        ← single source of truth for all lectures + tags
├── related-widget.js   ← injected into every lecture page
├── lectures/           ← processed lecture HTML files (generated by build.py)
└── scripts/
    └── build.py        ← copies + processes lectures, embeds catalog in index.html
```

**Never edit files inside `lectures/` by hand — they are overwritten by build.py.**
**The source lecture HTML files live in Google Drive (path in build.py).**

## Biomarkers — LOCKED DECISION (2026-06-06)

**Chosen model: Option B — typed edges, stored in a centralized `biomarkers` registry in `catalog.json`.**
This is locked. Do NOT add biomarkers as a flat 5th tag facet, and do NOT tag every
lecture that merely *mentions* a marker (40/45 lectures mention at least one — bare
mentions are noise). A biomarker connects to a lecture only through one of three typed
relationships:

- `primary` — the canonical lecture for understanding this marker. **Exactly one** lecture id.
- `driver` — lectures explaining a mechanism that *moves* the marker (upstream cause). 0+ ids.
- `predicts` — lectures that use the marker as a signal/outcome (downstream readout). 0+ ids.

There is deliberately **no `mentioned` edge** — if a marker only appears as passing
evidence, it gets no edge at all.

`biomarker_type` taxonomy (in `catalog.json → taxonomy.biomarker_type`):
`performance`, `lab`, `molecular_clock`, `composite`, `physiological`.

### biomarker registry schema

```json
"biomarkers": {
  "vo2_max": {
    "label": "VO₂ Max",
    "type": "performance",
    "primary": "lecture_vo2max_capacity_margin",
    "driver":   ["lecture_mitochondrial_dysfunction", "lecture_metabolic_inflexibility"],
    "predicts": ["lecture_white_matter_fitness", "lecture_autonomic_hrv"]
  }
}
```

Rules:
- The registry key (e.g. `vo2_max`) is a stable snake_case slug — never reuse or rename.
- `type` must be one of the `biomarker_type` taxonomy strings.
- `primary`, `driver`, and `predicts` ids must all be real lecture ids.
- A lecture id must NOT appear in more than one edge of the same biomarker (no overlap).
- A biomarker's `primary` lecture must actually discuss the marker (see validation keywords
  in `scripts/validate.py`).

### Validation

Run before every build/commit:

```bash
python3 scripts/validate.py
```

It checks tag correctness, schema completeness, related-link integrity, and the biomarker
registry (coverage, valid edges, no overlap, primary-lecture content). It exits non-zero on
any failure. `build.py` does not read the `biomarkers` block today, but it is embedded into
`index.html` with the rest of the catalog for future UI use.

## PII & publishing — LOCKED DECISION

**The lecture sources in Google Drive are intentionally personal.** The author
writes them to himself, with his own biomarker readings (hs-CRP, fasting insulin,
HOMA-IR, ApoB, TSH, vitamin D, …) and age framing ("in your fifties"). That is by
design and must stay in the sources.

**Nothing personal may be published.** `build.py` scrubs PII on the way from the
Drive sources into `lectures/`:

- `scripts/pii_rules.local.json` — **the only place the actual personal values
  live. It is gitignored and must never be committed** (committing the values in
  the scrub rules would itself be the leak). Holds `{pattern, repl, label}` rules
  that map each personal reading to an equivalent generic teaching range (e.g. a
  personal hs-CRP reading → `hs-CRP in the optimal range (<1.0 mg/L)`; a personal
  ApoB reading → `ApoB above the optimal range`).
- `scripts/build.py` — loads those rules + applies generic age-framing rules
  (`in your fifties` → `in midlife`, which contain no unique value).
- `scripts/validate.py` — the safety net. It loads the same local rules plus
  generic patterns (the old Gmail, the device hostname, and a second-person
  biomarker-reading heuristic `your <marker> … <number>`) and **fails the build**
  if any survive into a published file (`lectures/*.html`, `index.html`,
  `catalog.json`, `privacy.html`, `CLAUDE.md`, `scripts/build.py`).

**Workflow rule:** when a source lecture introduces a new personal value, add a
rule to `scripts/pii_rules.local.json` (never to the tracked code), then run
`python3 scripts/validate.py` before committing. The `PII_HEURISTIC` catches
second-person readings even without a rule. Never commit if the PII checks fail.
Never "fix" a PII failure by editing files in `lectures/` (they are regenerated)
or by hard-coding values into tracked files; fix the local rules instead. Do not
strip PII from the Drive sources — they are meant to stay personal.

## How to add new lectures (weekly workflow)

**Step 1 — Add the lecture HTML to Google Drive**
Save the new HTML file (e.g. `lecture_xyz.html`) in the DailyLearning Google Drive folder as usual.

**Step 2 — Add an entry to catalog.json**
Tell Claude: *"Add lecture_xyz.html to the catalog"*

Claude will:
1. Read the lecture HTML to extract the title and derive a one-sentence description
2. Assign appropriate tags across all four categories
3. Choose 3–4 intellectually related lectures from the existing catalog
4. Append a new entry to `catalog.json` following the schema below

**Step 3 — Run the build script**
```bash
cd ~/aging-library
python3 scripts/build.py
```

This copies the new lecture into `lectures/`, injects the related-articles widget, and re-embeds the catalog in `index.html`.

**Step 4 — Commit and push**
```bash
git add -A
git commit -m "Add lecture: [title]"
git push
```

GitHub Pages updates automatically within ~60 seconds.

## catalog.json schema

```json
{
  "id": "lecture_xyz",
  "title": "Short readable title",
  "description": "One punchy sentence — what makes this interesting to a curious reader.",
  "filename": "lectures/lecture_xyz.html",
  "added": "YYYY-MM-DD",
  "tags": {
    "mechanism":    ["One or two tags from mechanism list"],
    "system":       ["One to three tags from system list"],
    "theme":        ["One to two tags from theme list"],
    "intervention": ["Zero to three tags from intervention list"]
  },
  "related": [
    "lecture_id_1",
    "lecture_id_2",
    "lecture_id_3",
    "lecture_id_4"
  ]
}
```

Rules:
- `id` must match the HTML filename without the `.html` extension
- `filename` is always `lectures/<id>.html`
- `description` should be ≤ 30 words, written for curious non-experts
- `related` should list 3–4 lectures that are intellectually connected (not just topically similar)
- All tag strings must match the taxonomy above exactly (case-sensitive)

## How to update an existing lecture's tags or related links

Edit `catalog.json` directly, then re-run `python3 scripts/build.py`.

---

## Adding a new tag to the taxonomy

Rarely needed. If a new lecture genuinely doesn't fit existing tags:
1. Add the new tag string to the relevant array in `catalog.json → taxonomy`
2. Tag the new lecture with it
3. Backfill any existing lectures that should also have the new tag
4. Run build.py

## Local preview

```bash
cd ~/aging-library
python3 -m http.server 8000
open http://localhost:8000
```

The site requires a local server (not `file://`) so that the related-widget fetch works correctly.
Actually the related widget reads inline JSON, so `file://` works too — open `index.html` directly in a browser.

## GitHub Pages setup (one-time)

1. Create a new GitHub repo (e.g. `aging-library`)
2. Push this directory: `git remote add origin https://github.com/YOUR_USERNAME/aging-library.git && git push -u origin main`
3. In GitHub repo Settings → Pages → Source: `main` branch, `/ (root)` folder
4. Site will be live at `https://YOUR_USERNAME.github.io/aging-library/`
